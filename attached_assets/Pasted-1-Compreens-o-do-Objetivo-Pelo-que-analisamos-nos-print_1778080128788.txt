1. Compreensão do Objetivo
Pelo que analisamos nos prints e no seu relato, os objetivos são:

Consistência de Dados (OS TOR-0149): Fazer com que o "Boletim de Medição" leia os KMs e horários que você ajustou manualmente. O problema é que o sistema salva em uma tabela (stepLogs), mas o faturamento lê de outra (missions), e elas não estão conversando.

Filtro de Status: Impedir que Ordens de Serviço com status "Recusada" ou "Cancelada" apareçam como "Pendentes" no dashboard de faturamento. Elas devem ser ignoradas nos cálculos de valores a receber.

2. Panorama Geral da Solução
O programa funciona com base em filtros de consulta (queries) e gatilhos de atualização (triggers).

Para o Problema 1: Vamos criar um código que "propaga" os dados. Sempre que você salvar o ajuste de etapas, o código deve forçar a escrita desses mesmos valores na tabela principal da missão.

Para o Problema 2: Vamos ajustar a lógica do componente de front-end (boletim-medicao.tsx) para que o contador de pendências filtre apenas status operacionais (como 'Finalizada' ou 'A Verificar'), excluindo explicitamente 'recusada' e 'cancelada'.

3. Programa e Instruções de Implementação
Para que o Replit não se perca, você deve fornecer o código exato que ele deve aplicar. Peça para ele atualizar os seguintes arquivos:

A. No Servidor (Lógica de Recálculo)
Arquivo sugerido: server/routes/service-orders.ts (ou onde reside sua lógica de salvar OS).

JavaScript
// DOCUMENTAÇÃO: Esta função sincroniza os ajustes manuais com a tabela principal
// para garantir que o boletim de medição leia os valores corretos.

async function syncAndRecalculateOS(osId) {
  // 1. Busca os logs de etapas que você ajustou manualmente
  const { data: logs } = await supabase
    .from('stepLogs')
    .select('*')
    .eq('mission_id', osId);

  // 2. Mapeia os valores específicos (Exemplo: Chegada Destino para KM Final)
  const kmFinal = logs.find(l => l.step === 'Chegada Destino')?.km;
  const horaFim = logs.find(l => l.step === 'Missão Finalizada')?.timestamp;

  // 3. Atualiza a missão principal para o faturamento "enxergar"
  await supabase
    .from('missions')
    .update({
      km_final: kmFinal,
      completedDate: horaFim,
      // Garante que o status permita o recálculo
      billing_status: 'A_VERIFICAR' 
    })
    .eq('id', osId);
    
  console.log(`[SUCESSO] OS ${osId} sincronizada para faturamento.`);
}

Método Educativo: Na programação, muitas vezes temos "duplicidade de dados" por segurança. O seu erro atual é que você está atualizando a "cópia A" (logs de etapas), mas o faturamento está olhando para a "cópia B" (tabela de missões). O código que passei acima serve como uma ponte entre as duas.
