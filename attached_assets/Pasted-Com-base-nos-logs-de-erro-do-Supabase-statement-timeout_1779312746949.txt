O sistema TMSEGo está sofrendo de um gargalo de concorrência e IOPS. Quando uma operação pesada acontece (como ler arquivos de faturamento ou processar conciliações cruzando dados), o banco trava porque o Realtime mantém dezenas de conexões abertas escutando atualizações desnecessárias.

Com base na última inspeção, vamos aplicar as correções exatas nas frentes 1 e 2 para eliminar os timeouts do banco de dados. Pode executar os seguintes passos:

### 1. FRENTE 2 (Banco de Dados - SQL DDL)
Como você validou que alguns índices já existem, adicione ao arquivo `server/db-init.ts` (ou execute via script de migração) apenas a criação dos índices que estão faltando e são críticos para as chaves estrangeiras:

CREATE INDEX IF NOT EXISTS idx_escort_billings_client_id ON escort_billings(client_id);
CREATE INDEX IF NOT EXISTS idx_chat_messages_conversation_id ON chat_messages(conversation_id);
CREATE INDEX IF NOT EXISTS idx_chat_messages_sender_id ON chat_messages(sender_id);

### 2. FRENTE 1 (Otimização do Realtime no Frontend)
O canal global em `queryClient.ts` escutando mais de 25 tabelas em tempo real está gerando um efeito de "Fan-out" que derruba o pool de conexões do Postgres (`queue_timeout`). Vamos isolar o tráfego:

- Remova as tabelas de chat (`chat_messages`, `conversations`, `presence`) do canal global.
- Certifique-se de que as telas de chat (admin/chat, mobile/chat e o widget) gerenciem suas próprias assinaturas isoladas, limpando os canais (unsubscribe) quando os componentes forem desmontados.
- Se o canal global ainda estiver pesado, divida-o por escopo de módulo (ex: um canal dinâmico apenas para a operação ativa, separando dados estáticos ou relatórios).

### 3. NOTA SOBRE POLÍTICAS RLS (auth_rls_initplan)
Como as políticas RLS estão direto no Dashboard do Supabase e não no código local, não se preocupe em reescrevê-las agora. Deixaremos para otimizar as chamadas a `auth.uid()` direto pelo painel web do Supabase depois.
