1. Compreensão do Objetivo
O objetivo é criar uma função de sincronização que:

Leia a OS: Capture horários (início/fim), KMs e o status atual diretamente da tabela de missões.

Consulte o Cliente: Busque na tabela de clientes (ou contratos) a tabela de preços específica vinculada àquele CNPJ.

Gere o Boletim: Calcule o valor final e atualize o Boletim de Medição automaticamente sempre que a OS for salva ou alterada.

Pergunta de alinhamento: No seu banco de dados, qual é o nome da coluna na tabela de clientes que guarda os valores de preço (ex: valor_km, valor_hora) ou existe uma tabela separada chamada tabelas_precos?

2. Panorama Geral da Solução
O "travamento" acontece porque o Boletim de Medição provavelmente está tentando ler campos que ainda não foram calculados ou atualizados após o seu ajuste manual na OS.

Passos do desenvolvimento:

Trigger de Atualização: Vamos criar um gatilho que dispara sempre que você clica em "Salvar" na OS.

Busca de Preços: O sistema vai identificar quem é o cliente da OS e buscar os valores combinados (R$/KM, R$/Hora).

Lógica Financeira: O programa vai subtrair os horários e KMs para encontrar a base de cálculo.

Sincronização: O valor calculado será injetado diretamente no registro que o Boletim de Medição utiliza.

3. Programa e Instruções de Implementação
Aqui está o código completo para a função de sincronização. Você deve pedir para o Replit aplicar este bloco na sua API de serviços (backend).

JavaScript
/**
 * DOCUMENTAÇÃO: Sincronização OS -> Boletim de Medição
 * Este código garante que o financeiro siga a risca o que está na OS e no cliente.
 */

async function sincronizarOSComBoletim(osId) {
  try {
    // 1. Busca os dados atuais da OS (incluindo o ID do cliente)
    const { data: os, error: osError } = await supabase
      .from('missions')
      .select('*, client_id')
      .eq('id', osId)
      .single();

    if (osError || !os) throw new Error("OS não encontrada.");

    // 2. Busca a Tabela de Preços no cadastro do Cliente
    const { data: cliente, error: clientError } = await supabase
      .from('clients')
      .select('valor_km, valor_hora, taxa_base')
      .eq('id', os.client_id)
      .single();

    if (clientError) throw new Error("Tabela de preços do cliente não encontrada.");

    // 3. Lógica de Cálculo baseada estritamente na OS
    const kmTotal = (os.km_final || 0) - (os.km_inicial || 0);
    
    // Cálculo de tempo (Diferença entre horários da OS)
    const dataInicio = new Date(os.missionStartedAt);
    const dataFim = new Date(os.completedDate);
    const horasTrabalhadas = Math.abs(dataFim - dataInicio) / (1000 * 60 * 60);

    // 4. Cálculo Final
    const valorKM = kmTotal * (cliente.valor_km || 0);
    const valorTempo = horasTrabalhadas * (cliente.valor_hora || 0);
    const totalFinanceiro = valorKM + valorTempo + (cliente.taxa_base || 0);

    // 5. Atualiza o faturamento da OS para o Boletim ler o valor correto
    const { error: updateError } = await supabase
      .from('missions')
      .update({
        fat_total: totalFinanceiro,
        status_faturamento: 'A_VERIFICAR', // Move para a fila de aprovação
        sincronizado_em: new Date()
      })
      .eq('id', osId);

    if (updateError) throw updateError;

    console.log(`[TMSEGo] OS ${osId} sincronizada com sucesso. Valor: R$ ${totalFinanceiro.toFixed(2)}`);

  } catch (err) {
    console.error("[ERRO SINCRONIZACAO]:", err.message);
  }
}
Instruções detalhadas de implementação:
Vincule ao Botão Salvar: No seu frontend, onde você clica para salvar a OS, adicione a chamada para esta função sincronizarOSComBoletim(id).

Verifique os Campos: Certifique-se de que os nomes missionStartedAt e completedDate são exatamente os que você usa no banco.

Status Concluído: Como você mencionou que a OS está como "concluída", o código acima garante que, após o seu ajuste manual, ela seja marcada como A_VERIFICAR, entrando corretamente no fluxo de aprovação do Boletim de Medição.

Método Educativo: O que fizemos aqui foi garantir que o sistema não dependa de "leituras automáticas" instáveis. Agora, sempre que você altera um dado, o sistema recalcula tudo na hora e salva o resultado final. Assim, o Boletim de Medição não precisa calcular nada, ele apenas exibe o valor que já foi conferido e salvo na OS.
