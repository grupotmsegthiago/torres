### 1. FRENTE 2 (DATABASE DDL) - OUTPUT DO ADVISOR & LINTER

#### [LINTER 0001] Foreign Keys Sem Índice (Unindexed Foreign Keys)
O banco está realizando Full Table Scans para validar integridade referencial nas operações de escrita e atualização. É necessário criar índices nas seguintes chaves estrangeiras críticas do TMSEGo:

* **Tabela:** `escort_billings`
    * `client_id` -> aponta para `clients(id)`
    * `service_order_id` -> aponta para `service_orders(id)`
* **Tabela:** `service_orders`
    * `client_id` -> aponta para `clients(id)`
* **Tabela:** `chat_messages`
    * `conversation_id` -> aponta para `conversations(id)`
    * `sender_id` -> aponta para `auth.users(id)`

**Ação esperada no server/db-init.ts:** Gerar comandos `CREATE INDEX IF NOT EXISTS` concorrentes ou padrões para essas colunas de FK para estancar o Statement Timeout.

#### [LINTER 0003] RLS Reavaliada por Linha (auth_rls_initplan)
As políticas de Row Level Security (RLS) estão chamando `auth.uid()` ou subconsultas diretamente no predicado de checagem de linha. Isso força o Postgres a reavaliar a função para cada linha retornada (Complexidade O(N)).

* **Tabelas Afetadas:** `service_orders`, `escort_billings`, `clients` e `chat_messages`.
* **Problema Comum:** `CREATE POLICY ... ON table TO authenticated USING (auth.uid() = user_id);` ou subqueries dinâmicas.
* **Correção de Desempenho recomendada pela Supabase:** Reescrever as políticas injetando a checagem em variáveis de sessão estáveis ou transformando a query em um Join estático indexado, garantindo que o plano de execução (`InitPlan`) rode apenas uma vez por query e não por linha. Alternativamente, usar funções estáveis com `SECURITY DEFINER` onde for estritamente performático.
