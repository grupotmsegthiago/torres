# Torres Vigilância Patrimonial - Website & Sistema Interno

## Overview
This project delivers an institutional landing page and an internal management system for Torres Vigilância Patrimonial, a Brazilian security company. The system's core purpose is to streamline operational workflows, manage company assets, track personnel and vehicles, and enhance overall operational efficiency. It provides robust user and role management, detailed asset tracking (vehicles, weapons), automated operational processes (service orders, mission workflows), and real-time operational oversight. The strategic goal is to provide a comprehensive, integrated platform that supports company growth, ensures regulatory compliance, and elevates the quality of security services.

## ⛔ REGRAS CRÍTICAS — LEIA PRIMEIRO

### Diretrizes Obrigatórias de Desenvolvimento

1. **Idioma e Comunicação:** Código documentado em português. Variáveis e comentários em português quando possível.
2. **Verificação de Banco (Supabase):** Antes de qualquer alteração de lógica, verificar se as tabelas (financial_transactions, service_orders, mission_costs, etc.) existem e se os campos são persistentes. É proibido usar memória temporária ou localStorage para dados financeiros.
3. **Sincronismo Global (Realtime):** Qualquer atualização (Pedágio, Abastecimento, Status) deve usar `supabase.channel` para atualizar todas as abas abertas simultaneamente sem necessidade de F5.
4. **Prevenção de Erros Futuros:** O código deve ser escalável. A lógica de data (created_at) NÃO pode esconder custos de ontem no DRE de hoje. Custos de abastecimento devem ser persistentes até o fim da missão.
5. **Memória de Regras:** Ler este arquivo antes de cada commit. Se o código violar uma regra de negócio (como o pedágio ser custo e receita ao mesmo tempo), abortar e corrigir.
6. **Saúde do Sistema:** Não implementar soluções hardcoded (valores fixos no código). Tudo deve vir de consultas dinâmicas ao banco.
7. **Verificação Grid ↔ DRE:** Antes de finalizar qualquer alteração financeira, verificar se o dado reflete tanto no Grid Operacional quanto no DRE ao mesmo tempo.

### Regras de Dados e Acesso

- **O banco local do Replit (DATABASE_URL) NÃO é usado.** TODO acesso a dados usa Supabase — via REST API (`supabaseAdmin`) ou via conexão PostgreSQL direta (`SUPABASE_DATABASE_URL` com Drizzle ORM).
- **Data Access Paths:** `storage.*` methods use Supabase REST API (supabaseAdmin) with automatic snake_case↔camelCase conversion. Direct `db.*` calls in routes.ts use Drizzle ORM connected to `SUPABASE_DATABASE_URL`. Both hit the SAME Supabase PostgreSQL database.
- **NEVER add a `password` column** back to `shared/schema.ts` — authentication is handled entirely by Supabase Auth.
- **Always use `apiRequest()` or `authFetch()`** for API calls — never raw `fetch()`.
- **OS status values are stored with accents** (e.g., `"concluída"`) — always normalize before comparing.
- **`data_missao` must be stored as a full ISO timestamp** (e.g., `"2026-04-02T11:00:00"`) — never a date-only string (`"2026-04-02"`). PostgreSQL stores date-only strings as UTC midnight, which shifts the BRT date by one day back.
- **Fuso Horário Brasília (UTC-3):** Todo registro de data/hora e cálculos de duração devem operar em BRT (America/Sao_Paulo). O banco armazena timestamps sem timezone em UTC. No frontend, usar `parseUTCTimestamp()` para garantir parsing correto como UTC antes de calcular diferenças. Nunca usar `new Date(timestamp)` diretamente em timestamps sem 'Z' — isso interpreta como hora local do navegador.
- **`missionStartedAt` é setado no primeiro checkout** (checkout_armamento), não no em_transito_origem. Isso garante que o cronômetro reflita o tempo real da jornada completa.
- **Billing hora extra usa `missionStartedAt`** como início de cobrança. O cálculo de horas usa a RPC `calc_mission_elapsed_hours()` do banco (NOW() do Supabase) como fonte única de tempo. A função centralizada `calcularFaturamentoLive()` em `server/billing-calc.ts` é chamada pelo Grid, DRE, CRON e Dashboard. Hora Extra é RECEITA cobrada do cliente, nunca despesa de funcionário.
- **Sincronismo Total de Indicadores:** Todos os indicadores gerenciais (Dashboard, Balanço, Metas, Grid) devem ser re-calculados dinamicamente com base nos dados brutos das missões, garantindo 100% de paridade com os DREs individuais. É proibido ter valores diferentes em telas diferentes.
- **CRON billing** MUST use `supabaseAdmin.from("service_orders")` (REST API with snake_case columns) — NEVER `storage.getServiceOrders()`. Only recalculates active missions; concluded missions are never touched.

### Regras de Persistência Financeira

- **Custos operacionais devem ser calculados dinamicamente no banco** (`mission_costs` table) e refletidos via Realtime no Grid e DRE. É proibido usar campos totalizadores estáticos ou cache de servidor que gerem dessincronia.
- **Custo de abastecimento = mission_cost vinculado à OS.** Toda vez que um fueling é registrado (mobile ou admin), o backend cria um `mission_cost` (category "Combustível", cost_type "expense") vinculado à OS ativa daquela viatura. O Grid e DRE leem custos via `getMissionCostsByOS()` — fonte única de verdade. O `vehicleFuelCache` (financial_transactions) é apenas fallback quando não há mission_costs. Startup sync (`syncFuelingMissionCosts`) garante que fuelings anteriores sejam backfillados.
- **Custo de abastecimento é persistente.** O Grid/DRE deve mostrar TODOS os abastecimentos da viatura durante a missão (soma total), não apenas o mais recente. O valor deve permanecer no DRE da missão até que ela seja finalizada, independente da virada de data à meia-noite. NUNCA filtrar combustível por `due_date = TODAY()`.
- **Pedágio com missão = Custo + Receita (Reembolso).** Se `service_order_id` tiver valor, o sistema cria AMBOS os registros (expense + revenue) em `mission_costs` e `financial_transactions`. Impacto zero no lucro.
- **Pedágio sem missão (Vazio) = apenas Despesa.** Se `service_order_id` for null, o valor entra apenas como despesa da viatura (abate o lucro direto). Categoria: "Custos Fixos/Deslocamento Extra".
- **Valores financeiros são imutáveis após gravação.** Uma vez que um custo foi atribuído a uma missão, ele deve ser gravado como valor fixo. O sistema não deve recalcular ou filtrar esse valor pela data atual do servidor.

## System Architecture
The system employs a modern web stack: React with TypeScript and Vite for the frontend, and Express with Supabase Auth for the backend. **Supabase (PostgreSQL) is the ONLY database**. Tailwind CSS is utilized for styling, maintaining a professional monochrome aesthetic (black/white) with Montserrat/Inter typography. UI components adhere to an "Enterprise UI" design, ensuring a consistent and polished user experience.

**Key Architectural Decisions & Implementations:**
-   **Authentication & Authorization:** JWT-based authentication via Supabase Auth, implementing robust Role-Based Access Control (RBAC) through a `perfis_acesso` table. Session cookies are not used.
-   **Data Layer:** Two access paths to the SAME Supabase PostgreSQL:
    1. `storage.*` (in `server/storage.ts`) — uses Supabase REST API (`supabaseAdmin`) with `toSnakeObj()`/`toCamelObj()` converters. Returns camelCase TypeScript types.
    2. `db.*` (in `server/routes.ts`, `server/db-init.ts`) — uses Drizzle ORM connected to `SUPABASE_DATABASE_URL`. Returns camelCase via schema mapping.
    3. `supabaseAdmin.from(...)` — direct REST API calls in CRON and some routes. Returns snake_case.
-   **Operational Grid:** A central feature providing real-time vehicle tracking, status monitoring (speed, idle, signal loss), and integration with risk management companies.
-   **Mobile Interface for Field Agents:** A dedicated mobile-first experience for `funcionario` roles, supporting mission workflows, checklists, GPS tracking, and photo capture.
-   **Smart OCR Integration:** Utilizes OpenAI Vision for intelligent data extraction from documents (CNH, CNV, weapon registrations) to auto-fill forms.
-   **Vehicle Tracking Integration:** Real-time vehicle positions and telemetry are sourced from the TrucksControl / NewRastreamentoOnline SOAP webservice.
-   **API Design:** All API endpoints are defined in `server/routes.ts`, covering comprehensive CRUD and operational functionalities across various entities.
-   **UI/UX Design System:** A professional black/white aesthetic, specific typography, and standardized UI elements ensure visual consistency. The `DIRETORIA` role is visually highlighted.
-   **Google Maps Integration:** Used for location services, autocomplete, and mapping in the operational grid and escort calculation.
-   **Automated Tasks:** Cron jobs manage fleet monitoring and HR compliance checks.
-   **Login Selfie Enforcement:** `funcionario` users must take a selfie upon login, enforced by a protected route and stored in the `login_selfies` table.
-   **Audit System:** Comprehensive tracking of vigilante actions, page views, and security events (e.g., screenshot attempts, tab visibility changes) with an admin dashboard for monitoring and alerting.
-   **Financial Module:** Manages accounts payable/receivable with transaction CRUD, installment support, reporting, and category breakdown. **Automatic Financial Integration:** `financial_transactions` now has `origin_type` (manual/escort_billing/fueling/maintenance/mission_cost) and `origin_id` columns. Auto-transactions are created when: escort billing is APROVADA (INCOME), fueling is registered (EXPENSE), maintenance is registered with cost (EXPENSE), mission costs are added (EXPENSE). Auto-transactions are removed when the source record is deleted. Auto-transactions are locked — cannot be edited or deleted manually via the API or frontend. The Relatório tab includes a "Saldo Realizado" card and auto-transaction count. API: `GET /api/financial/resumo` returns comprehensive breakdown by origin type.
-   **Escort Calculation Engine:** Manages escort billing, client-specific price tables, frequent routes, and generates numbered Mission Bulletins (Boletim de Missão) with detailed calculation rules (KM, minimums, nocturnal/hazardous additions).
-   **OS Logic (Ordem de Serviço):** Implements the complete OS processing flow:
    - **Status Flow:** Funcionário fills OS data → status "A_VERIFICAR" → Administrativo reviews → "APROVADA" (auto-generates BO) or "REJEITADA" (solicita correção com motivo).
    - **Horário de Início Rule:** `Inicio_Missao = max(Horario_Agendado, Horario_Chegada_Real)` — if vigilante arrives early, scheduled time is used; if late, real arrival time is used.
    - **Franchise-Based Billing:** KM excedente is calculated as `max(0, KM_Carregado - Franquia_Cliente)`. Value breakdown shows franchise vs excess KM charges.
    - **Auto-Submit on Mission Close:** When a mission reaches "encerrada" status, the system automatically creates an `escort_billings` record with status "A_VERIFICAR", pulling KM from mission photos and times from mission lifecycle.
    - **Admin Review Panel:** In Financeiro > Boletim Medição tab, pending OS ("A Verificar") are displayed with full summary (horário considerado, horas trabalhadas, KM, franquia, excedente, valor total) and Aprovar/Rejeitar buttons.
    - **API Endpoints:** `POST /api/escort/billings/submit-os` (mobile submit), `POST /api/escort/billings/:id/revisar` (admin review), `GET /api/escort/billings/pendentes` (pending list).
    - **DB Columns Added:** `horario_agendado`, `horario_inicio_considerado`, `horas_trabalhadas`, `km_franquia`, `km_excedente`, `valor_franquia`, `valor_km_extra`, `revisado_por`, `revisado_em`, `motivo_rejeicao`, margin/result fields.

Selecting a client opens their "pasta" with 6 tabs: **Veículos** (client vehicles with CRUD + auto-registration from missions), **Tabelas** (price tables + frequent routes per client), **Contratos** (service contracts with validity/signature control), **Relatório de Missões** (completed missions with period filter), **Relatório de Faturamento** (billing summary with open vs closed boletins), and **Homologação** (document dispatch for client onboarding).

Features: company document upload/management (`company_documents` table), selectable document checkboxes, optional Apresentação Institucional and Tabela de Valores, SMTP configuration (env vars or per-request), HTML email template with Torres branding, and send history log (`homologation_logs` table). Auto-upsert: when a field agent fills the escorted vehicle plate during a mission, the plate (and driver info) is automatically registered/updated in `client_vehicles`.     - **30-Second Auto-Sync:** Operational grid, vehicle tracking, and financial transactions refetch every 30 seconds (`refetchInterval: 30000`) to prevent stale "ghost" data.
    - **Fueling KM Validation:** Both admin and mobile fueling endpoints reject KM entries with >1500km jump from current odometer, preventing data entry errors that distort consumption averages.
-   **Mission Updates & Agent Reports:** Multi-step update flow for field agents: (1) photo capture, (2) message with smart context-aware suggestions (e.g., "Chegada na Origem" only shows within 2km geofence), (3) send to admin. Photos stored as base64 data URLs in `mission_updates.photo_url` (validated server-side, max 5MB, must be `data:image/*`). Operational grid shows agent's last update message/step in blue badge. "Atualização Recente" button on vehicle cards copies a formatted report template (WhatsApp-style with emojis: OS number, status, date/time, client, origin/destination, driver info, agents, progress %, occurrence, location, Google Maps link) to clipboard along with the agent's photo. `mission_updates` table: `id, service_order_id, os_number, employee_id, employee_name, message, mission_step, latitude, longitude, photo_url, read_by_admin, created_at`. Alert banner polls unread updates every 15s. State preserved on send failure for retry.

-   **Mobile Folha de Ponto (Clock In/Out):** Full time tracking from mobile at `/mobile/ponto`. 4-step process: Entrada → Saída Almoço → Retorno Almoço → Saída. Each step requires photo capture (front camera) and GPS location. Data stored in `employee_timesheets` with `clock_in_photo/lat/lng`, `clock_out_photo/lat/lng`, `lunch_out_photo/lat/lng`, `lunch_in_photo/lat/lng` fields. API: `GET /api/mobile/ponto/today`, `POST /api/mobile/ponto/clock`.
-   **Mobile Abastecimento (Fueling):** Mobile fueling registration at `/mobile/abastecimento`. Requires 3 mandatory photos: bomba, NF, hodômetro. Enter KM, liters, R$/litro, posto. Auto-calculates total. Validates KM > current vehicle KM. Updates vehicle KM on submit. Returns oil change alert if applicable. **Financial sync (3-way linkage):** on successful registration: (1) `createAutoTransaction()` creates a `financial_transaction` (EXPENSE, category "Combustível", origin_type "fueling") for financial dashboard; (2) if the vehicle has an active OS, creates a `mission_cost` record linked to that OS (category "Combustível", cost_type "expense") so the DRE sees the fuel cost immediately; (3) the DRE detail endpoint prevents double-counting — when `mission_costs` has fuel entries, the `financial_transactions` fueling lookup is skipped. Frontend calls `invalidateRelatedQueries("vehicle"/"financial"/"mission-cost")` which cascades to all related query keys AND propagates to other open tabs via BroadcastChannel. API: `GET /api/mobile/abastecimento/vehicle`, `POST /api/mobile/abastecimento`. Schema additions: `vehicle_fueling.pump_photo`, `vehicle_fueling.odometer_photo`, `vehicle_fueling.latitude/longitude`, `vehicle_fueling.gasoline_price`, `vehicle_fueling.ethanol_price`, `vehicle_fueling.fuel_recommendation`, `vehicle_fueling.recommendation_followed`.
-   **Mobile Ocorrências (Incident Reports):** Mobile incident reporting at `/mobile/ocorrencia`. Types: Acidente, Quebra Mecânica, Avaria, Manutenção, Segurança, Outro. Supports up to 5 photos + description + GPS. History tab shows past reports with admin responses. API: `GET/POST /api/mobile/ocorrencias`, `GET /api/ocorrencias` (admin), `PATCH /api/ocorrencias/:id` (admin). Table: `employee_occurrences` (employee_id, vehicle_id, type, description, photos[], lat/lng, status, admin_notes).
-   **Ponto Operacional (Long Duration Time Clock):** Extended time tracking for field agents on multi-day trips. Mobile page at `/mobile/ponto-operacional` allows agents to open a shift and keep it running indefinitely (across days) until they click "Finalizar Ponto". Features: live chronometer, entry/exit with optional notes, monthly accumulated hours display. Admin dashboard at `/admin/ponto-operacional` (sidebar: Funcionários > Ponto Operacional) shows all agents with: progress bar (green < 190h, amber 190-220h, red > 220h), "Em Viagem" badge for agents with open shifts, expandable per-agent history with individual records. 220h monthly limit rule: hours above 220h classified as overtime with 50/50 split (50% payroll, 50% employee bonus). Salary base: R$ 2,432.50. Table: `ponto_operacional` (Supabase-only, UUID PK) with employee_id, employee_name, entrada, saida, horas_decimal, status (aberto/fechado), observacao. API: `GET /api/ponto-operacional/aberto`, `POST /api/ponto-operacional/entrada`, `POST /api/ponto-operacional/saida`, `GET /api/ponto-operacional/resumo-mensal?mes=YYYY-MM`, `GET /api/ponto-operacional/historico/:employeeId`, `DELETE /api/ponto-operacional/:id`.
-   **Oil Change Alert:** Vehicles track `last_oil_change_km`. System warns at 9,000 km since last change and alerts at 10,000 km. Alerts shown on fueling submission and via `GET /api/mobile/oil-alert/:vehicleId`.
-   **Reference Points (Pontos de Referência):** Admin can register named geographic reference points on the map via right-click context menu on any vehicle. Each point has a name, coordinates (from vehicle's current position), and configurable radius (default 500m). When any vehicle enters a reference point's radius, the "Referência" column in the vehicle table shows the point name. Reference point circles are drawn on the map with colored borders. Full CRUD via `GET/POST/PATCH/DELETE /api/reference-points`. Table: `reference_points` (id, name, latitude, longitude, radius_meters, color, created_at). Management dialog accessible from context menu.
-   **Vehicle Context Menu (Right-Click):** All vehicle actions moved from the "Ações" button column to a right-click context menu on vehicle table rows AND map markers. Actions: Localizar no Mapa, Visualizar Pré-Alerta, Espelhamento, Enviar Comando (bloquear/desbloquear/sirene/aviso cabine/mensagem), Retornar à Base, Liberar Início Antecipado, Copiar Relatório, Nova OS, Docs VTR, Finalizar Missão, Cancelar Missão, Cadastrar Ponto de Referência, Gerenciar Referências. The "Ações" column was replaced by "Referência" column.
-   **Extended Price Table (escort_contracts):** New fields: `valor_acionamento` (R$/KM acionamento), `franquia_horas`, `franquia_km`, `valor_hora_extra`, `valor_km_extra`, `valor_cancelamento`, `custo_deslocamento_100km` (rodando com saída da base). Displayed in client Tabelas tab with 2-row card layout.
-   **Agent Location Tracking (Rastreamento de Agentes):** Automatic background GPS tracking of field agents every 10 minutes. The mobile app (`MobileLayout`) requires GPS permission on login and sends position updates via `POST /api/agent/location`. Current positions stored in `agent_locations` (upsert per user). Full position history stored in `agent_location_history` with `(user_id, created_at)` index for efficient date queries. Admin panel at `/admin/tracker` (Rastreador in sidebar under Frota) shows real-time agent positions with online/offline status (15min threshold), Google Maps links, accuracy/speed data, and per-agent daily history dialog with date picker. Read endpoints (`GET /api/agent/locations`, `GET /api/agent/locations/:userId/history`) are admin-only (`requireAdminRole`). Auto-refresh every 60 seconds on admin panel.
-   **Mission Route Tracking (Rastreamento de Rota):** Records vehicle GPS positions during active missions (status `em_andamento`) in the `mission_positions` table. Positions are stored with 50m minimum distance filter using an in-memory map to avoid DB reads on each poll cycle. Endpoints: `GET /api/service-orders/:id/positions` (raw positions), `GET /api/service-orders/:id/route` (planned route polyline from Google Directions API + positions, cached in OS `route` field). Frontend: "Ver Rota" button in map info window and vehicle context menu. Draws planned route (black polyline), traveled route (blue polyline), start marker (green dot), last position marker (red dot), origin/destination arrows on the operational grid map. Floating legend bar with close button to dismiss. Uses `google.maps.geometry.encoding.decodePath` for polyline rendering. Window events (`show-route`) for cross-component communication.
-   **Route Progress Bar (Barra de Progresso GPS):** Visual progress indicator on grid cards and forward dialog showing vehicle position between origin and destination. Uses `getRouteProgress()` (haversine-based with 1.3x road multiplier) to calculate % complete and remaining distance. `RouteProgressBar` component renders a route line with car icon, origin (blue) / destination (red) dots, percentage, and "X km restantes" label. Compact variant on grid cards, full variant in forward/share dialog. Falls back to step-based `getMissionProgress()` when GPS coordinates unavailable. WhatsApp reports include `{{routeInfo}}` template variable with distance details.
-   **TrucksControl Resilience (Last Valid Position):** Three improvements to vehicle tracking reliability: (1) **Empty packet handling** — messages with lat=0/lon=0 are rejected from overwriting valid positions; system maintains `lastValidMessageByVehicle` and `lastValidMessageBySpy` maps as fallback. (2) **Timestamp ordering** — incoming messages sorted by tracker's `dt` timestamp (not API `mId`), preventing buffer dumps from reordering positions chronologically. (3) **Valid coord validation** — `hasValidCoords()` validates lat/lon are non-zero and within geographic bounds before storing. When a vehicle temporarily loses GPS, the grid continues showing the last known valid position instead of "Sem sinal".

## Real-Time Data Synchronization
-   **Query Cache:** `staleTime: 30s` + `refetchOnWindowFocus: true` ensures data refreshes automatically when switching between pages/tabs.
-   **Centralized Invalidation:** `invalidateRelatedQueries(scope)` in `client/src/lib/queryClient.ts` provides cross-page cache invalidation. Scopes: `"vehicle"`, `"employee"`, `"billing"`, `"financial"`, `"service-order"`, `"mission-cost"`. Each scope cascades invalidations to all related queryKeys (e.g., changing a vehicle also invalidates dashboard, operational grid). **Supabase Realtime** channel (`realtime-sync-*`) listens for INSERT events on `mission_costs`, `financial_transactions`, `vehicle_fueling`, and `mission_updates` tables + UPDATE on `service_orders` to auto-invalidate caches across all open tabs/devices. Channel uses exponential backoff reconnection (up to 30s delay, max 10 retries). On `online` event, all channels are re-subscribed and queries invalidated.
-   **Financial Transaction Sync:** Billing approval creates financial transactions with `origin_type: "escort_billing"`. The approval flow removes any legacy `"service_order"` origin transactions before creating the correct one. Total = `fat_acionamento + fat_hora_extra + fat_km + despesas_pedagio`.

### Conexão Infalível (Agent App Resilience)
- **É proibido deslogar o usuário por inatividade ou expiração de token sem tentativa automática de renovação.** A integridade da comunicação entre App e Painel Admin é prioridade máxima.
- **Token de Longa Duração:** Supabase Auth emite access_token (~1h) + refresh_token (~30 dias). O app renova o token proativamente a cada 45 minutos via `supabase.auth.refreshSession()`. Se um request retorna 401, tanto `apiRequest` quanto `offlineQueue.flushQueue` tentam refresh antes de falhar.
- **Persistência Offline (v2):** Ações offline são enfileiradas em `localStorage` via `offlineQueue.ts`. Max 12 retries com backoff exponencial (5s→30s + jitter). Flush imediato ao enfileirar (800ms). Padrão reativo `subscribeQueue` com `flushResult.eventId` edge-triggered evita toasts duplicados. Botão "Reenviar" manual no banner de sincronização. `isNetworkError` expandido para 502/503/504 (janela de deploy).
- **Reconexão Agressiva:** Quando detectado offline, um watchdog tenta reconectar a cada 5 segundos (via `/api/health` ping). Ao reconectar: refresh do token + invalidação de todas as queries. Banner visual mostra "Reconectando..." com spinner.
- **WakeLock Keep-Alive:** `navigator.wakeLock.request("screen")` solicitado no mount do app do agente. Re-solicitado ao retornar de background (visibilitychange → visible). Previne o OS de dormir a tela durante missão.
- **Visibilitychange Re-ping:** Ao voltar para a aba/app, o agente faz imediatamente: ping `/api/health` + `supabase.auth.refreshSession()`. Se refresh falha, reporta para `POST /api/auth/token-failure`.
- **Token Failure Logging:** Backend `POST /api/auth/token-failure` registra falhas de refresh na tabela `token_failure_logs` (employee_id, employee_name, error_message, user_agent, ip_address, created_at). Permite auditoria de problemas de autenticação em campo.

### Ouvinte Imortal (Admin Grid Resilience)
- **Monitoramento de Conexão Admin:** O Grid detecta `online`/`offline` events e exibe banner vermelho "Tentando reconectar..." quando offline. Ao reconectar, faz invalidação total de queries (Fetch Imediato).
- **Supabase Realtime Push (Imortal):** Canal Realtime escuta `mission_updates`, `service_orders` (INSERT+UPDATE+*), `vehicle_fueling`, `mission_costs`, `financial_transactions`, `escort_billings`. Reconexão infinita (sem cap de retries), heartbeat a cada 60s que reconecta se stale >120s ou disconnected. Re-subscribe automático no `visibilitychange` (volta de background) e `online` event. Scope `mission-update` adicionado para invalidar grid+updates+service-orders em cascata.
- **Auto-refresh global:** Intervalo reduzido de 60s para 30s. Polling do grid operacional mantém 30s via `REFRESH_INTERVAL_MS`.
- **Payload otimizado:** `/api/mission/updates` e `/api/operational-grid` não retornam mais fotos base64 inline — substituem por `[has_photo]` + `hasPhoto: true`. Fotos carregam on-demand via `GET /api/mission/updates/:id/photo`. Resposta do grid caiu de ~5-20s para <1s.
- **Alerta VTR Offline (5min):** Se uma viatura com missão ativa não envia update por >5 minutos, o card dela muda para VERMELHO com badge "SEM SINAL Xmin" piscante e ícone WifiOff. Um alarme sonoro distinto (3 bipes curtos em onda quadrada, 660/440Hz) é disparado quando uma VTR recém-entra em estado offline. Debounce de 30s para evitar alarmes repetitivos.
- **Alerta VTR Crítico (15min):** Se uma viatura passa >15 minutos sem update, o card escala para vermelho intenso (ring-4, shadow-lg, bg-red-100) com badge "PERDA DE SINAL Xmin" + ícone WifiOff. Alarme crítico `playCriticalAlarm()` dispara 5 bipes (880/660/440Hz, volume 0.35) — mais alto e mais longo que o alarme padrão. Debounce de 30s independente.
- **Som de Notificação:** Beep curto (880Hz sine wave) ao receber qualquer nova atualização de agente. Toggle persistido em localStorage.

-   **Selo de Auditoria (Copy Audit Stamp — Produtividade Livre):** Copy buttons ("Copiar Texto", "Copiar Foto") ALWAYS execute the copy immediately — never block, never show `alert()` or `confirm()`. The stamp "(Copiado por: [Nome] às [DD/MM HH:MM])" is purely informational, displayed below the copy buttons. Each click: (1) copies to clipboard first, (2) shows "Relatório Copiado!" toast, (3) fires `POST /api/mission/updates/:id/copy-audit` in background to update stamp. Stamp syncs in real-time via Supabase Realtime UPDATE listener. The audit log is for history only and must NEVER impede subsequent clicks. Schema: `mission_updates.copiado_por` (text), `mission_updates.copiado_em` (timestamp).

### Segregação de Funções (Security Level 1)
- **Barreira Web para Agentes:** `ProtectedRoute` em `App.tsx` verifica `user.role === "funcionario"` e redireciona para tela "Acesso Negado" (`access-denied.tsx`). Agentes só acessam `/mobile/*`. Login por CPF no `/admin` redireciona automaticamente para `/mobile`.
- **Blindagem Financeira (API):** Todas as rotas `/api/financial/*`, `/api/escort/billings/*`, `/api/escort/contracts`, `/api/employees/:id/salary-*` exigem `requireAdminRole` (role `admin` ou `diretoria`). Agentes recebem 403 Forbidden.
- **Omissão no App Mobile:** App de abastecimento mostra apenas dados operacionais (litros, km, relação etanol/gasolina). Nenhum dado de faturamento, lucro, margem ou DRE é acessível via rotas mobile.
- **REGRA INVIOLÁVEL:** É terminantemente proibido exibir dados financeiros para perfis de agentes. O acesso web é restrito a administradores. Agentes operam exclusivamente via App Mobile.

### Identidade Tática (Tactical Identity)
- **Reverse Geocoding:** `GET /api/reverse-geocode?lat=X&lng=Y` usa Nominatim (OpenStreetMap) para converter coordenadas GPS em endereço completo (rua, número, bairro, cidade/UF). Cache client-side em `_reverseGeocodeCache`.
- **Componente `ReverseGeocodeAddress`:** Exibe endereço completo com ícone `ShieldCheck` (verde esmeralda) + link "Ver no mapa". Usado no feed de notificações (MissionUpdatesAlert) e na AlertsTimeline.
- **Ícone Shield:** Nas atualizações de missão, `ShieldCheck` substitui `MapPin` para reforçar identidade tática/segurança patrimonial.
- **Nominatim Rate Limit:** 1 req/seg. Cache evita chamadas duplicadas. Endereços formatados: "Rua X, 123, Bairro, Cidade/UF".
